# Setup Guide & Migration Summary

This document explains all changes made to the timekeeping system, how to set it up, and how the new architecture works.

## What Changed

### Backend Architecture Overhaul

**Old Structure:**
```
com.timekeeping.api/
├── controller/      (all controllers together)
├── service/         (all services together)
├── repository/      (all repositories together)
├── entity/          (all entities together)
├── config/
├── dto/
└── security/
```

**New Structure (Layered + Event-Driven):**
```
com.timekeeping/
├── domain/          Domain objects, repository interfaces, domain events
├── application/     Use-cases, commands, results, event handlers
├── infrastructure/  Spring Data adapters, JWT, security details
├── presentation/    REST controllers, HTTP DTOs
├── config/          Spring configuration
└── shared/          Exception hierarchy
```

### Key Improvements

1. **Domain-Driven Design**
   - `domain/` layer contains pure business logic
   - Repository interfaces in domain (not JPA-dependent)
   - Domain events: `UserRegisteredEvent`, `TimeInRecordedEvent`, `TimeOutRecordedEvent`

2. **Event-Driven Architecture**
   - When a user registers → `UserRegisteredEvent` published
   - When user clocks in → `TimeInRecordedEvent` published
   - When user clocks out → `TimeOutRecordedEvent` published
   - `AuditEventHandler` listens and logs asynchronously
   - New side-effects can be added without modifying services (Open/Closed principle)

3. **SOLID Principles**
   - **S**: Each class has one responsibility (JWT handling, audit logging, business logic)
   - **O**: New behaviors via event listeners, not modifying existing code
   - **L**: Adapters properly implement domain repository interfaces
   - **I**: Service interfaces are focused and separate
   - **D**: Services depend on domain interfaces, not concrete JPA repositories

4. **Production-Grade Code**
   - `GlobalExceptionHandler` with structured error responses
   - Input validation on all request DTOs
   - Async event processing doesn't block requests
   - Proper logging throughout
   - Non-root Docker user

### Dependency Changes

**Removed:**
- Lombok (Java 25 compatibility issues per user preference)

**Added:**
- Spring Boot Actuator (health endpoints)
- Flyway (database migrations)
- Spring Boot 4.0.3 (from 3.5.5)

**Updated:**

### Frontend Changes

**React 18 → React 19:**
- Removed `import React from 'react'` from all JSX files (automatic JSX runtime)
- Updated `@vitejs/plugin-react` to v4.3.0
- Updated `vite` to v6.0.0
- Updated `react-router-dom` to v7.0.0
- Updated `date-fns` to v4.1.0

**API Response Changes:**
- `POST /api/timelog/time-in` now returns `TimeLogDto` directly (not `{success, timeLog}`)
- `POST /api/timelog/time-out` now returns `TimeLogDto` directly
- Status values now uppercase: `PRESENT`, `HALF_DAY`, `ABSENT` (was lowercase)

**Frontend Components Updated:**
- `Timesheet.jsx` — fixed API response handling for timeIn/timeOut
- `AttendanceCalendar.jsx` — handles enum statuses, converts to CSS classes
- All components removed `import React` (automatic in React 19)

## Database

**Changed from MongoDB to PostgreSQL:**
- Uses PostgreSQL 16
- Flyway manages migrations
- `V1__create_schema.sql` creates `users` and `time_logs` tables

**Key Points:**
- UUIDs generated by PostgreSQL (`gen_random_uuid()`)
- Timestamps default to `NOW()`
- Foreign key constraint from time_logs → users
- Unique constraint on (user_id, date) in time_logs

## Docker

**Three-stage setup:**

1. **PostgreSQL 16-alpine**
   - Healthcheck ensures DB is ready before app starts
   - Data persisted in named volume `postgres_data`

2. **Backend (Java 25 Spring Boot)**
   - Multi-stage build: `eclipse-temurin:25-jdk-alpine` (build) → `eclipse-temurin:25-jre-alpine` (runtime)
   - Runs as non-root user (`appuser`)
   - Environment variables from `.env`
   - Exposed on port 5000

3. **Frontend (React + Nginx)**
   - Multi-stage build: `node:22-alpine` (build) → `nginx:stable-alpine` (serve)
   - nginx.conf handles SPA routing and proxies `/api/` to backend
   - Caches static assets aggressively (1-year expiry)
   - Exposed on port 80

## Setup Instructions

### Prerequisites

**For Local Development:**
- Java 25
- Maven 3.9+
- PostgreSQL 16
- Node.js 22+

**For Docker:**
- Docker & Docker Compose

### Option 1: Docker Compose (Recommended)

```bash
# Clone or navigate to project root
cd timekeeping

# Copy environment template
cp .env.example .env

# Edit .env and set:
#   DB_PASSWORD=your_secure_password
#   JWT_SECRET=your_64_char_secret_key

# Build and start
docker compose up --build

# Wait for postgres healthcheck (10-15 seconds)
# Frontend will be available at http://localhost
# Backend at http://localhost:5000
```

**Troubleshooting:**
```bash
# View logs
docker compose logs -f

# Stop all services
docker compose down

# Remove volumes (WARNING: deletes data)
docker compose down -v

# Rebuild a specific service
docker compose build --no-cache backend
docker compose up -d backend
```

### Option 2: Local Development

**Backend:**
```bash
# 1. Install Java 25 and Maven 3.9+
java -version     # Should show 25.x
mvn --version     # Should show 3.9.x

# 2. Ensure PostgreSQL is running locally
# On macOS: brew services start postgresql@16
# On Ubuntu: sudo systemctl start postgresql
# On Docker: docker run -d --name postgres -e POSTGRES_PASSWORD=postgres postgres:16-alpine

# 3. Create database
createdb -U postgres timekeeping

# 4. Build and run backend
cd backend
mvn clean spring-boot:run
# Backend now runs on http://localhost:5000
```

**Frontend:**
```bash
cd frontend
npm install
npm run dev
# Frontend runs on http://localhost:3000
# Automatically proxies /api calls to http://localhost:5000
```

**Access:**
- Frontend: http://localhost:3000
- Backend: http://localhost:5000
- Health: http://localhost:5000/actuator/health

### Environment Configuration

**`.env` file (root):**
```env
# Database
DB_USER=postgres
DB_PASSWORD=changeme

# JWT (REQUIRED — generate a strong value for production)
JWT_SECRET=change-this-to-a-secure-random-64-char-secret
JWT_EXPIRATION=2592000000

# Ports (optional)
BACKEND_PORT=5000
FRONTEND_PORT=80
```

**Spring Profiles:**
- Local: `application.properties` (connects to `localhost:5432`)
- Docker: `application-docker.properties` (connects to `postgres:5432` service)

Automatically selected via `SPRING_PROFILES_ACTIVE=docker` in docker-compose.yml

## Frontend Integration Notes

**Authorization:**
- All protected endpoints require `Authorization: Bearer <token>` header
- Frontend automatically adds this via axios interceptor in `src/services/api.js`
- Token stored in localStorage as `token`

**Attendance Status Display:**
- Backend returns uppercase: `PRESENT`, `HALF_DAY`, `ABSENT`
- Frontend converts to CSS-friendly class: `present`, `half-day`, `absent`
- `statusLabel()` function in AttendanceCalendar.jsx maps to human-readable labels

## Database Migrations

To add a new migration:

1. Create a file: `backend/src/main/resources/db/migration/V2__description.sql`
2. Write your SQL:
   ```sql
   -- Example: Add a new column
   ALTER TABLE time_logs ADD COLUMN approved_by UUID REFERENCES users(id);
   ```
3. Flyway automatically applies it on next application startup
4. Check logs to confirm: `Flyway migration successful`

## Performance Tuning

**Backend:**
- Virtual threads enabled (Java 21+ feature)
- Connection pool size: 10 max, 2 min (configurable in properties)
- Event handlers run async (non-blocking)

**Frontend:**
- Static assets cached 1 year
- Vite performs tree-shaking and minification

**Database:**
- Indexes on `user_id`, `date`, `(user_id, date)` in time_logs
- UUID generation at DB level
